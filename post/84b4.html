Spring与事务管理

Spring对事物管理的支持

事务，指数据库中的一个操作序列，它由一系列的SQL指令组成。

Spring在高层次建立事务抽象，然后在此基础上提供一个统一的编程模型，这意味着Spring具有在多种环境中配置和使用事务的能力。

基于SpringIOC和AOP，Spring提供声明式事务管理的方式，他匀速开发人员直接在Spring容器中定义事务的边界和属性。

Spring还实现了事务管理和数据访问的分离，开发人员只需关注对当先事务的界定，其余工作将由Spring框架自动完成。

事务管理的核心接口

Spring主要通过三个接口实现事务抽象：TransactionDefinition、TransactionStatus、PlatformManager

它们都位于org.springframework.transaction包中。

TransactionDefinition接口

该接口主要定义事务的属性，这些属性包括事务的隔离级别，事务的传播行为，事务的超时时间，是否为只读事务等。

事务的隔离级别

事务的隔离级别主要是指事务之间的交叉作用，这里首先要了解以下事务的几种隔离级别：

  1. 脏读

    A事务执行过程中，B事务读取了A事务修改后的数据。但是由于某些原因，A事务可能没有完成提交，发生RollBack了操作，则B事务所读取的数据就会是不正确的。这个未提交数据就是脏读（Dirty Read）。

  2. 不可重复读

    B事务读取了两次数据，在这两次的读取过程中A事务修改了数据，B事务的这两次读取出来的数据不一样。B事务这种读取的结果，即为不可重复读（Nonrepeatable Read）。

  3. 幻读

    B事务读取了两次数据，在这两次的读取过程中A事务添加了数据，B事务的这两次读取出来的集合不一样。这个流程看起来和不可重复读差不多，但幻读强调的集合的增减，而不是单独一条数据的修改。

TransactionDefinition针对于此定义了五种隔离级别，具体如下表所示：

隔离级别 描述
ISOLATION_DEFAULT 采用当前数据库默认的隔离级别
ISOLATION_READ_UNCOMMITTED 允许读取尚未提交的数据变更，可能会导致脏读、幻读和不可重复读
ISOLATION_READ_COMMITTED 允许读取已经提交的数据变更，可以避免脏读，无法避免幻读和不可重复读
ISOLATION_REPEATABLE_READ 允许数据重复读，可以避免脏读，不可重复读，不能避免幻读，资源消耗上升
ISOLATION_SERIALIZABLE 事务串行执行，可以避免脏读、幻读、不可重复读，资源消耗最大

事务的传播行为

事务的传播行为是指事务处理过程中所跨越的对象将以什么样的方式参与事务

TransactionDefinition定义了其中事务传播行为，具体如下表所示：

传播行为 描述
PROPAGATION_REQUIRED 默认的事务传播行为，如果当前存在一个事务，则加入该事务，如果当前没有事务，则创建一个新事物
PROPAGATION_SUPPORTS 如果当前存在一个事务，则加入该事务，如果当前没有事务，则直接执行
PROPAGATION_MANDATORY 当前必须存在一个事务，如果没有，就抛出异常
PROPAGATION_REQUIRED_NEW 创建一个新事物，如果当前已存在一个事务，则将已存在的事务挂起
PROPAGATION_NOT_SUPPORTED 不支持事务，在没有事务的情况下执行，如果当前已存在一个事务，则将已存在的事务挂起
PROPAGATION_NEVER 永远不支持当前事务，如果当前已存在一个事务，则抛出异常
PROAGATION_NESTED 如果存在当前事务，则在当前事务中的一个子事务中执行

Spring中声明式事务对传播行为以来较大，开发人员可根据实际需要选择使用

事务的超时时间和是否只读

事务的超时时间指事物执行的时间界限，超过这个时间界限，事务将会回滚。TransactionDefinition接口提供了TIMEOUT_DEFALUT的常量定义，用来指定事物的超时时间。

当事物的属性为只读时，该事务不修改任何数据，只读事务有助于提升性能，如果在只读事务中修改数据，可能会引发异常。

TransactionDefinition接口

TransactionDefinition接口提供了一系列方法来获取事物的属性，具体如下表所示：

方法 描述
IntgetPropagationBehavior() 返回事务的传播行为
int getIsolationLevel() 返回事务的隔离层次
boolean isReadOnly() 判断事务是否为只读
String getName() 返回定义的事务名称
int getTimeout() 返回事务的超时属性

上表列举了TransactionDefinition接口提供的方法，程序可通过TransactionDefinition接口的这些方法获取当前事务的属性。

TransactionStatus接口

TransactionStatus接口主要用于界定事务的状态，通常情况下，编程式事务中使用该接口较多。

TransactionStatus接口中提供了一系列返回事务状态信息的方法，具体如表所示：

方法 描述
boolean isNewTransaction() 判断当前事务是否为新事务
boolean hasSavepoint() 判断当前事务是否创造了一个保存点
boolean isRollbackOnly()0 判断当前事务是否被标记为roolback-only
void setRollbackOnly() 将当前事务标记为rollback-only
boolean isCompleted() 判断当前事务是否已经完成（提交或回滚）
void flush() 刷新底层的修改到数据库

上表列举了TransactionStatus接口提供的方法，事务管理器可以通过该接口 提供的方法获取事务运行的状态信息，也可以使用setRollbackOnly()方法间接回滚事务。

PlatformTransactionManager接口

PlatformTransactionManager接口是Sprin事务管理的中心接口，它真正执行了事务管理的职能，并针对不同的持久化技术封装了对应的实现类，具体如下表所示：

TransactionStatus getTransaction(Transaction Definition definition) 根据事务定义获取一个已存在的事务或创建一个新的事务，并返回这个事务的状态
void commit(TransactionStatus status) 根据事务的状态提交事务
void rollback(TransactionStatus status) 根据事务的状态回滚事务

编程式事务管理

编程式事务管理使用一个模板类TransactionTemplate，该类位于org.springframework.transaction.support包中，TransactionTemplate类封装了与PlatformTransactionManager相关的事物界定操作，开发人员可以通过对应的callback()接口实现具体内容的界定。

方法 描述
void serTransactionManager(PlatformTransactionManager transactionManager) 设置事务管理器
Platform TransactionManager getTransactionManager() 返回事务管理器
T execute(TransactionCallback<T>action) 在TransactionCallback接口中定义需要以事务方式组织的数据访问逻辑
void rollbackOnException(TransactionStatus status, Throwable ex) 处理事务遇到异常时调用的方法
