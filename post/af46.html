Spring理念简介

Spring最成功的是其提出的理念，而不是技术本身，Spring所依赖的核心理念有两个：

  1. 控制反转（Inversion of Control，IoC）；
  2. 面向切面编程（Aspect Oriented Programing，AOP）。

本章首先介绍IoC思想。

基于注解的IoC

由于Spring Boot是基于注解的开发，因此下文仅介绍全注解的方式讲述Spring IoC技术。

IoC：Inversion of Control，控制反转，是一种通过描述来生成或获取对象的技术，这个技术不是Spring或Java独有的，而是一种理念。对于Java来说，我们常用new关键字来创建对象，而在Spring中则通过描述（XML文件或注解）来创建或者获取对象。

一个系统可以生成各种对象，并且这些对象都需要进行管理。由于对象间很可能存在各种依赖关系（一对多，如班级和学生；多对多，如学生和老师；等），为此Spring还提供了依赖注入的功能，是我们能够通过描述来管理各个对象之间的关系。

为了管理各种对象，我们需要一个容器，在Spring中把每一个需要管理的对象称为Spring Bean（简称Bean），该容器被称为Spring IoC容器（简称为IoC容器）。

IoC容器需要具备两个基本的功能：

  1. 通过描述管理Bean，包括发布和获取Bean；
  2. 通过描述完成Bean之间的依赖关系。

IoC容器简介

通过观察源码，我们能看到该接口含有多个getBean方法，这些方法是IoC容器最重要的方法，他允许我们通过类型或者名称获取Bean，这对后面讲讲到的Spring的依赖注入（Dependency Injection，DI）是十分重要的。

在默认情况下，Bean都是以单例形式存在的，也就是使用getBean返回的都是同一个对象。与单例相反，若一个Bean对象的isPrototype方法返回值为True，则不是单例模式，每次使用getBean时，容器会创建一个新的Bean返回给调用者，这与后面讨论的Bean的作用域相关。

由于BeanFactory的功能还不够强大，因此Spring在BeanFactory的基础上，还设计了更高级的ApplicationContext，它是BeanFactory的子接口之一，在Spring体系中，BeanFactory和ApplicationContext是最重要的两个接口设计，实际使用中大部分IoC容器都是ApplicationContext的接口实现类。它们的关系如下图所示：

如何装配Bean

装配Bean的方式有两种：

  1. 基于配置类的方式
  2. 基于注解的装配方式

常用的方式是基于注解的装配方式，下面会分别介绍这两种方式的做法。

基于配置类的装配方式

基于设置类的装配方式主要分为三部分：

  1. 创建POJO类
  2. 在配置类中创建装配方法
  3. 创建ApplicationContext对象获取Bean对象

创建POJO类

对于项目中的Java类，我们称之为POJO类（Plain Ordinary Java Object，简单Java对象），因此我们先定义一个这些POJO类。注意这里需要getter和setter方法。

在根目录下创建一个pojo包，专门放置POJO实体类，本例创建User类。

package io.gitee.yunchongxiao.chapter03.pojo;

public class User {
    private Long id;
    private String userName;
    private String note;

    public Long getId() {
        return id;
    }

    public void setId(Long id) {
        this.id = id;
    }

    public String getUserName() {
        return userName;
    }

    public void setUserName(String userName) {
        this.userName = userName;
    }

    @Override
    public String toString() {
        return "User{" +
                "id=" + id +
                ", userName='" + userName + '\'' +
                ", note='" + note + '\'' +
                '}';
    }

    public String getNote() {
        return note;
    }

    public void setNote(String note) {
        this.note = note;
    }
}

在配置类中创建装配方法

首先需要编写一个配置类，在根目录下创建config包，在改包下创建AppConfig.java文件，该文件就是配置类，在类名上用@Configuration将该类注解为配置类。

为配置类编写装配方法，每一个方法对应一个Bean的注册，在方法上用@Configuration将该方法对应为一个Bean。Spring根据该装配类生成IoC，并根据类中的方法装配Bean。

package io.gitee.yunchongxiao.chapter03.config;

import io.gitee.yunchongxiao.chapter03.pojo.User;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class AppConfig {

    @Bean(name = "user")
    public User initUser() {
        User user = new User();
        user.setId(1L);
        user.setUserName("Green");
        user.setNote("Where is the Alice");
        return user;
    }
}
  • @Configuration：该注解代表这是一个Java配置文件，Spring容器会根据它来生成IoC容器，并装配Bean
  • @Bean：该注解代表讲initUser方法返回的POJO装配到IoC容器中，而其属性name定义这个Bean的名称，若没有配置它，则讲方法名称“initUser”作为Bean的名称保存到Spring IoC容器中。

通过ApplicationContext对象获取Bean

我们在测试类中对该Bean进行获取测试，将AppConfig.class传入AnnotationConfigApplicationContext对象中，生成ApplicationContext对象。

通过ApplicationContext.getBean方法可以获取Bean对象，可以通过类型进行获取，可以通过名称进行获取（默认为方法名，本例中设置为了”user“）。

package io.gitee.yunchongxiao.chapter03;

import io.gitee.yunchongxiao.chapter03.config.AppConfig;
import io.gitee.yunchongxiao.chapter03.pojo.User;
import org.junit.jupiter.api.Test;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.context.ApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

@SpringBootTest
class DemoApplicationTests {

    @Test
    void contextLoads() {
    }

    @Test
    void IoCTest() {
        ApplicationContext appContext = new AnnotationConfigApplicationContext(AppConfig.class);
        User user = appContext.getBean(User.class);
        // User user = appContext.getBean("user");
        System.out.println(user);
    }

}

基于注解的装配

如果使用一个个@Bean注解进行装配，那将是一件很麻烦的事情，由于Spring Boot是基于注解的方式，因此下面主要介绍基于注解的方式来介绍Bean的装配。

基于注解的装配有两个要点：

  1. 通过@ComponentScan注解设置扫描包路径
  2. 通过@Value注解初始化对象

设置扫描包路径

我们在Appconfig.java这个配置类中通过@ComponentScan设置扫描的对象包路径，这里是pojo包。

@ComponentScan中有两种设置方式：

  1. 正则匹配，如：@ComponentScan(“io.gitee.yunchongxiao.chapter03.*”)
  2. 指定基础包，如：@ComponentScan(basePackages = {“io.gitee.yunchongxiao.chapter03.pojo”})
  3. 什么都不指定，即@ComponentScan，则扫描该类所在的包及其子包。

个人比较喜欢第二种指定基础包的形式，仁仁智智吧

package io.gitee.yunchongxiao.chapter03.config;

import io.gitee.yunchongxiao.chapter03.pojo.User;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.Configuration;

@Configuration
@ComponentScan(basePackages = {"io.gitee.yunchongxiao.chapter03.pojo"})
public class AppConfig {

    @Bean(name = "user")
    public User initUser() {
        User user = new User();
        user.setId(1L);
        user.setUserName("Green");
        user.setNote("Where is the Alice");
        return user;
    }
}

创建POJO类

注意，这里的POJO类为了能够被扫描到，需要添加＠Component注解，这样一来，所有pojo包下的@Component注解的对象都会注册到IoC容器中，对象通过@Value注解进行赋值。

package io.gitee.yunchongxiao.chapter03.pojo;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

@Component("player")
public class Player {
    @Value("0")
    private Long id;
    @Value("Alice")
    private String name;
    @Value("444049406")
    private String password;

    public Long getId() {
        return id;
    }

    public void setId(Long id) {
        this.id = id;
    }

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public String getPassword() {
        return password;
    }

    public void setPassword(String password) {
        this.password = password;
    }

    @Override
    public String toString() {
        return "Player{" +
                "id=" + id +
                ", name='" + name + '\'' +
                ", password='" + password + '\'' +
                '}';
    }
}

依赖注入

依赖注入是指，对象之间存在嵌套关系，如：Person类中含有Animal类，对于这样的Person类，就无法使用上述方法进行Bean注入，这时候我们就需要依赖注入方法了。

例如，人类（Person）有时候利用动物（Animal）完成一些事情，如让狗（Dog）看家，或者让猫（Cat）变身，下面我们就让Person类拥有一个Cat。

这里采取扫描包路径的方式，即上述的第二种方式进行Bean注入。

创建Animal接口类

这里为什么不直接创建一个Cat类呢，因为我们有可能会遇到这种存在继承关系的对象，这时候在依赖注入时会有一点问题，这里就直接使Cat类和Dog类直接继承Animal类，方便后续的讲解。

package io.gitee.yunchongxiao.chapter03.definition;

public interface Animal {
    public void active();

}

这里仅定义了方法，属性在实现类中再定义（其实是不知道在抽象类中定义的属性如何进行初始化赋值）。

创建Cat实体类

Cat类继承了Animal类，同时为自己设置了一个name属性，这里用@Value注解进行初始化即可。

package io.gitee.yunchongxiao.chapter03.pojo;

import io.gitee.yunchongxiao.chapter03.definition.Animal;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

@Component
public class Cat implements Animal {
    @Value("Chocolate")
    private String name;
    @Override
    public void active() {
        System.out.println("🐱：喵~");
    }

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }
}

生命周期

Bean的生命周期，通常分为Bean定义、Bean初始化、Bean的生存期和Bean的销毁四部分组成。

Spring通过我们的配置，如@ComponentScan定义的扫描路径扫描带有@Component的类，对这些类进行信息的解析，并将定义的信息保存下来，存放到IoC容器中，这就是Bean的定义（注意这里并没有对Bean进行初始化）。通常的，在Bean的定义完成后，IoC容器会接着进行Bean的初始化，即对IoC中的Bean进行初始化、依赖注入等操作。

  • 即使定义了ApplicationContextAware 接口，有时候也不会调用，这要根据定义的IoC 容器来决定。 Spring IoC 容器最低的要求是实现BeanFactory接口，而不是实现ApplicationContext 接口。对于那些没有实现ApplicationContext接口的容器，在生命周期对应的ApplicationContextAware 定义的方法也是不会被调用的，只有实现了ApplicationContext接口的容器，才会在生命周期调用pplicationContextAware 所定义的setApplicationContext方法。
  • 我们所编写的IoC容器大多都实现了ApplicationContext，所以不存在这个问题。

    private Animal animal;

    }

    @Override
    public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
        System.out.println("【" + this.getClass().getSimpleName() + "】调用ApplicationContextAware的setApplicationContext ");
    }

    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("【" + this.getClass().getSimpleName() + "】调用InitializingBean的afterPropertiesSet方法");
    }

    @Override
    public void destroy() throws Exception {
        System.out.println("【" + this.getClass().getSimpleName() + "】调用DisposableBean方法");
    }

    @PostConstruct
    public void construct() {
        System.out.println("【" + this.getClass().getSimpleName() + "】注解@PostConstruct定义的自定义初始化方法");
    }

    @PreDestroy
    public void destruct() {
        System.out.println("【" + this.getClass().getSimpleName() + "】注解@PreDestroy定义的自定义销毁方法");
    }
}

可以发现，仅创建IoC容器后，Bean就会进行这么多的操作（在这里，我们仅对Neet类进行了接口的实现，并没有对Animal类进行这些接口的实现，所以依赖注入虽然在第一位，但是Animal类的这些操作是隐含在这步之前的）

细心的大佬可能已经发现了，我们并没有实现BeanPostProcessor，因为它是针对所有Bean都生效的接口，所以不能在某个特定的Bean中实现该功能，而应该单另一个类：

package io.gitee.yunchongxiao.chapter03.life;

import org.springframework.beans.BeansException;
import org.springframework.beans.factory.config.BeanPostProcessor;
import org.springframework.stereotype.Component;

@Component
public class BeanPostProcessExample implements BeanPostProcessor {
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
        System.out.println("BeanPostProcessor调用"
                + "postProcessBeforeInitialization方法，参数【"
                + bean.getClass().getSimpleName() + "】【" + beanName + "】");
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
        System.out.println("BeanPostProcessor调用"
                + "postProcessAfterInitialization方法，参数【"
                + bean.getClass().getSimpleName() + "】【" + beanName + "】");
        return bean;
    }
}

可以看到，该类对所有的Bean都生效了。
